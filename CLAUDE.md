# 記帳 / 庫存後台 App 開發規則

## 專案定位

- 本專案是手機維修店使用的營運後台
- 不是單純記帳 app
- 核心功能包含：訂單、工單、庫存、進貨、支出、報表、商品管理
- 所有修改必須以「手機維修店實際營運」為優先

## 核心開發規則

- 不可重構整個專案
- 不可重新設計架構
- 不可擅自修改 UI
- 不可修改顏色、排版、字體、按鈕樣式
- 只能針對我指定的問題做最小修改
- 不可新增我沒有要求的功能
- 不可刪除既有功能
- 不可簡化既有資料結構
- 修改前先說明會動到哪些檔案與 function

## 資料保護規則

- 不可破壞 localStorage 既有資料
- 不可任意更改 key 名稱
- 必須保護以下資料：
  - 訂單資料
  - 工單資料
  - 進貨資料
  - 庫存資料
  - 商品分類
  - 支出資料
  - 報表資料
  - 會員資料
- 修改資料結構前，必須提供舊資料相容邏輯
- 危險操作前必須提醒備份

## 庫存規則

- 庫存異動必須有紀錄
- 工單使用零件時要扣庫存
- 取消工單時要補回庫存
- 盤點補正要記錄原因
- 員工漏記、遺失報廢、盤點誤差要分開記錄
- 通用型號邏輯不可破壞
- 保護貼、鏡頭貼可共用型號
- 充電線不可自動繼承手機型號
- 庫存低於安全數量時需要警示

## 成本 / 利潤規則

- 利潤 = 收款金額 - 零件成本 - 必要費用
- 現金不可扣 Line Pay 手續費
- 只有 Line Pay 才能扣手續費與營業稅
- 零件成本必須獨立顯示
- 盤點誤差、損耗、補正成本必須能追蹤
- 商品移轉若有價差，必須記錄差額
- 報表需同步反映成本、損耗、盈虧

## 訂單 / 工單規則

- 新增訂單後畫面要回到最上方
- 訂單金額不可自動亂變
- 付款方式會影響利潤計算
- 工單使用的零件要能追蹤使用時間
- 訂單取消不能直接刪除庫存紀錄
- 訂單狀態修改要避免影響歷史報表

## 商品管理規則

- 商品管理要保持乾淨
- 新增商品應使用小按鈕、彈窗或收合區
- 不可讓大型新增表單直接佔滿主畫面
- 商品需支援品項、品牌、系列、型號
- 商品刪除按鈕必須可正常點擊
- 商品分類邏輯需清楚可維護

## 報表規則

- 報表要一眼看懂
- 圖表不可過大
- 手機版不可跑版
- 付款方式文字不可被切掉
- 營業額、零件成本、利潤要分開顯示
- 年度、月份、每日報表計算邏輯需一致

## 回覆與修改方式

- 優先使用繁體中文
- 回覆直接，不要過度冗長
- debug 時先分析問題來源
- 優先輸出 diff
- 不要直接重寫整份檔案
- 不要輸出簡化版
- 不要省略重要程式碼

## Git / Deploy Rules

- 每次完成修改後，直接建立 PR 並 merge 到 main
- 不要停留在 claude/* 分支
- main 是正式部署分支
- 修改完成後不需要再詢問我是否 merge
- merge 前必須確認登入、訂單、庫存、報表、商品管理沒有被破壞

## Context / Clear Rules

- 當對話過長或 context 混亂時，主動提醒我使用 /clear
- 完成一個功能後，建議我 /clear 再繼續下一個功能
- /clear 後仍必須完整遵守 CLAUDE.md 所有規則
- context 不足時先分析現有程式，不可亂猜
